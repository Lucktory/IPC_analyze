# Migrations

Order matters. Run each file in the Supabase SQL editor (or via `psql`) in the order below. Each migration is idempotent (`if not exists` / `or replace`) so re-running them is safe.

## Order

| # | File | Date | Purpose |
|---|---|---|---|
| 0 | `db/schema.sql` | — | Initial full schema. Run once on a fresh database. |
| 1 | `db/migration-2026-06-09-contract-period-notes.sql` | 2026-06-09 | Adds `contract_period_notes` table for per-(contract, period) notes written from the contract detail page. |
| 2 | `db/migration-2026-06-09-property-landlords.sql` | 2026-06-09 | Adds `property_landlords` junction so a property can have multiple owners independent of any active contract. |
| 3 | `db/migration-2026-06-11-banks-fields.sql` | 2026-06-11 | Adds operational fields to `banks`: `monthly_fee`, `transfer_fee_pct`, `transfer_fee_fixed`, `contact_name`, `contact_phone`, `contact_email`, `notes`. |
| 4 | `db/migration-2026-06-11-pending-actions-sent.sql` | 2026-06-11 | Adds `pending_actions_sent` snooze table for the Pendientes "Marcar como enviado" workflow (7-day snooze). |
| 5 | `db/migration-2026-06-13-liquidacion-grid.sql` | 2026-06-13 | Adds `contracts.lfa_code` (admin handler) + `liquidaciones.adjustment_amount` (signed manual adjustment) — feeds the 18-column liquidación grid. |
| 6 | `db/migration-2026-06-13-contract-commission.sql` | 2026-06-13 | Adds `contracts.commission_pct` (default 8.0%) for the auto-commission generator. |

## Other SQL files in `db/`

| File | When to run |
|---|---|
| `db/rls-disable.sql` | **Dev only.** Disables row-level security on every table. Already applied to dev. Production deployment must NOT apply this — RLS policies need to be designed first. |
| `db/seed-from-sheet.sql` | Optional. Generated by `npm run import-sheet` from `client-data/alejandro-sheet.csv`. Run after schema + migrations if you want to backfill historical data. |

## Deployment checklist

When deploying to a new Supabase project:

1. Run `db/schema.sql`.
2. Run all migrations 1 → 6 in order.
3. (Optional) `npm run import-sheet` then run `db/seed-from-sheet.sql` to load historical data.
4. **Do NOT** run `db/rls-disable.sql` on production.
5. Verify `auth.users` is set up; create initial admin user via Supabase dashboard.
6. Set `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` on Vercel.

## Verifying state

```sql
-- All migrations applied?
select column_name from information_schema.columns
 where table_name = 'contracts'
   and column_name in ('lfa_code', 'commission_pct');
-- Should return both rows.

select column_name from information_schema.columns
 where table_name = 'liquidaciones'
   and column_name = 'adjustment_amount';
-- Should return one row.

select table_name from information_schema.tables
 where table_schema = 'public'
   and table_name in (
     'contract_period_notes',
     'property_landlords',
     'pending_actions_sent'
   );
-- Should return all three.
```

## Schema-source convention

Migrations modify the live database without rewriting `db/schema.sql` — but every migration is also reflected back into `db/schema.sql` for documentation. So if you `cat db/schema.sql` you see the **current full schema** as of the latest migration. Treat `db/schema.sql` as the source of truth for the *final state*, and the migration files as the *deltas* applied to get there.
